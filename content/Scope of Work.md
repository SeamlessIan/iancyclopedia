A Scope of Work is a document which is completed in advance of beginning work on a project. The purpose of this document is to create in advance a written description of the details of the project.

Preparing a Scope of Work will pay off in a number of ways:
- It makes your job easier once you begin work on the substance of the project, since you have a roadmap that you built in advance
- It eliminates any chance of ambiguity - you want your expectations and those of the project owner to be exactly the same.

## The Scope Statement

### Mission
- This is the most high-level description of the project.
- The mission, or goal, for the project - what is the project?
- Summarize the project in one sentence.
### Objectives
- Why is it important to achieve the project mission ?
- What do you hope to achieve by executing the project?
- How will the business processes or customer experience be different once the project is complete?
### Deliverables
- What are the specific items that will be created and delivered to the client at the end of the project?
- This is a list of verifiable items. You'll know when you're done with the project when each one is complete.
- During the final walkthrough of the project with the client, you can iterate through each item on this list to  demonstrate completion.
### Stakeholders, roles, and responsibilities
#### Explanation
- This section names members of the client's and your own team who will be involved in the project.
- Here you can specify the involvement that the client will need to have during the project.

#### Example
- {Client Name}: Project Owner
  - Responsibilities:
    - Review and approve the project scope before Developer starts working
    - Attend Deliverable walkthrough and confirm completion of deliverables
    - Settle invoice within 2 weeks of receipt
- Ian Melchior: Developer
  - Responsibilities:
    - Write the Scope of Work and get Project Owner's approval
    - Complete the deliverables
    - Conduct deliverable walkthrough with Project Owner
    - Send invoice

### High-level work breakdown and structure
#### Explanation
- This section outlines the milestones and steps that will be met throughout the course of the project.
- Creation of a good work plan here provides a reassuring road map to the client, and saves the developer time by planning out the work required in advance.

#### Example
1. Project Owner explains requirements to Developer
2. Developer creates Scope of Work
3. Project Owner reviews & approves Scope of Work
4. Developer makes changes to the application according to the Scope of Work
5. Project Owner and Developer walk through the new application and confirm that all Deliverables are completed
6. Developer sends invoice to Project Owner
7. Project Owner settles invoice within 2 weeks

### Assumptions
#### Explanation
- This is a "CYA" section.
- You are making a million assumptions since you both live in a reality which brought you together for this project, but here, you want to specifically name the ones that could cost you or the client money.
#### Example
- All project requirements are contained within this Scope document.
- If a feature is unable to be completed using the capabilities of the chosen platform, Developer and Project Owner will try to find a suitable workaround without changing the terms of the agreement.
- Project Owner is familiar with the chosen platform's costs and limitations

### Communications
#### Explanation
- This section specifies when and how you'll communicate with the client throughout the course of the project.
#### Example
- Most communication will be via email.  Online meetings and phone calls can be scheduled as needed.
- Deliverable walkthrough will be conducted in-person or via screenshare session
  - If existing application:
    - Developer makes changers to live version of Project Owner's application, OR
    - Project Owner sends *.ds file to Developer, after which *DS file is sent back to Project Owner when all Deliverables have been satisfied
    - Developer chooses which option to use
  - If new application:
    - Developer builds application on developer account, then transfers .ds file to Project Owner after completion of the deliverable walkthrough

### Risks
#### Explanation
- If there's anything you simply can't know until the project is underway, name those items here
- Advise the client on things (s)he should be aware of once taking over the final product
#### Example
- The app may contain sensitive data which must be protected by Project Owner

### Documentation
#### Explanation
{Where will project documentation be kept? How will it be kept up-to-date?}

#### Example
- Documentation will be created in a shared Google document, shared from Developer to Project Owner, OR
- Updates on features will be communicated via email

### Terms
#### Explanation
- When and how will the project owner take ownership of the final product?
- When and how will you get paid?
#### Example
- The total cost of the project is ${Total Cost}, plus any expenses incurred in the development of the project.
- The work to be done includes all deliverables mentioned above, as well as any consultation and planning sessions already completed by Developer with Project Owners, and the creation of this Scope of Work.
- Developer will provide support for the deliverables outlined in this document for 30 days after the project is delivered. 
- Invoice will be issued by Developer after the deliverable review meeting. 
- Project Owners will issue payment within 15 days of receipt of the invoice
